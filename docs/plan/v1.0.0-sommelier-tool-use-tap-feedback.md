# v1.0.0 — Sommelier tool-use & header tap feedback

## Overview

Two small changes, version-bumped to 1.0.0 to mark the app as feature-complete
for the original scope.

1. **Sommelier robustness** — replace text-based JSON output with Claude's
   `tool_use` so apostrophes / smart quotes / mixed delimiters in producer or
   wine names can no longer break parsing.
2. **Header tap feedback** — give the *Add wine* and *Settings* header buttons
   a red `border-wine` ring while pressed, matching the existing type-filter
   pattern. Visible feedback for screen recordings.

Also drops the "include at least 2 of each type available in the cellar"
instruction from the sommelier prompt; the Querschläger requirement stays.

---

## Feature 1: Sommelier tool-use

### Problem

The sommelier occasionally fails with a JSON parse error such as
`expected '}' but got "'"` when the model's response mixes single and double
quotes around values that contain apostrophes (e.g. `Domaine d'Auvenay`,
`L'Étoile`). The previous implementation regex-extracted the first
`{...}` block from the text response and ran `JSON.parse` on it.

### Approach

Use the Anthropic Messages API tool-use feature. Define a single tool
`provide_recommendations` whose `input_schema` is the recommendations shape,
and force `tool_choice: { type: 'tool', name: 'provide_recommendations' }`.
The API then returns the data already parsed in `content[].input` — no regex,
no `JSON.parse`, no quoting drift possible.

### Implementation

- `app/src/lib/sommelier/claude.ts`:
  - Add `TOOL` constant describing the input schema (recommendations array
    with producer/name/vintage/reason/querschlaeger/drinkNow).
  - Send `tools: [TOOL]` and `tool_choice` on the request.
  - Replace text-extract + `JSON.parse` with a `find` for the `tool_use`
    block and read `block.input`.
  - Drop the JSON-format example from the system prompt; replace with
    "Use the provide_recommendations tool to return the result."
  - Drop the "include at least 2 of each type" line from the prompt.

- `app/src/lib/sommelier/claude.test.ts`:
  - Mock responses use `{ type: 'tool_use', name: 'provide_recommendations',
    input: ... }`.
  - Add a regression test for apostrophes in producer / name.
  - Assert `tool_choice` is forced on the request.

### Acceptance criteria

- [x] Sommelier responses parse cleanly even when names contain apostrophes
- [x] No `JSON.parse` on free-form model text anywhere in the sommelier path
- [x] All existing sommelier tests pass; new apostrophe regression test passes
- [x] Querschläger requirement preserved
- [x] No type-variety constraint in the prompt

## Feature 2: Header tap feedback

### Problem

When recording the iPhone screen for a blog post, taps on header buttons
(*Add wine*, *Settings*) leave no visible trace. The bottle-type filters
already show a red ring when active; the header buttons did not.

### Approach

Match the type-filter pattern: `rounded-full border-2 border-transparent`
plus an `active:border-wine` variant. The transparent baseline keeps layout
stable; the red ring shows for the duration of the tap.

### Implementation

- `app/src/routes/+page.svelte` — add the four classes to the *Add wine*
  `<a>` and the *Settings* `<a>` in the header.

### Acceptance criteria

- [x] Tapping *Add wine* shows a red ring before navigation
- [x] Tapping *Settings* shows a red ring before navigation
- [x] No layout shift when ring appears
- [x] No regression to the type-filter ring styling
