# v0.9.10 — Session-persistent filters & Sommelier

## Overview

Two features:
1. **Session-persistent filters** — remembering the active filter when navigating to a wine detail page and back
2. **Sommelier** — AI-powered wine pairing suggestions from the user's cellar

---

## Feature 1: Session-persistent filters

### Problem

When the user sets a filter (e.g. "white"), taps into a wine detail, then taps "Back", the filter resets to defaults. The `← Back` link navigates to `resolve('/')` without preserving filter state. URL params are only read on mount; they aren't set when filters change.

### Approach

Use `sessionStorage` to persist all five filter values (`activeType`, `activeProducer`, `activeCountry`, `activeSort`, `bottleFilter`). Write on every filter change, read on mount (with URL params taking priority if present, to preserve the "More from this producer" deep-link behavior). Session storage clears automatically when the app is closed, matching the "per-session" requirement.

### Implementation

1. Create `app/src/lib/data/session-filters.ts`:
   - `saveSessionFilters(state)` — writes filter state to `sessionStorage` as JSON
   - `loadSessionFilters()` — reads and parses, returns `null` if missing/corrupt
   - Key: `kellerverwaltung-session-filters`

2. In `app/src/routes/+page.svelte`:
   - On mount: load session filters first, then override with URL params if present
   - Add `$effect` that saves filter state to session storage whenever any filter value changes

### Acceptance criteria

- [ ] Setting a type filter, navigating to wine detail, pressing Back restores the filter
- [ ] Setting producer/country/bottle/sort filters and navigating back restores all of them
- [ ] URL params (e.g. "More from this producer") override session filters
- [ ] Opening the app fresh (new tab/session) starts with default filters
- [ ] Unit tests for `saveSessionFilters` and `loadSessionFilters`

## Feature 2: Sommelier

### Problem

The user wants quick wine pairing suggestions from their own cellar, described in plain text (e.g. "focaccia with chicken shawarma and oven legumes").

### Approach

Add a "Sommelier" button on the KPI line (right-aligned, opposite the bottle count text). Tapping it opens a modal/overlay with a text input and a submit button. On submit, call the Claude API (same Haiku model and API key already stored for OCR) with:
- The user's plain-text prompt describing what they want to pair with
- The **entire wine cellar** (all wines with producer, name, type, vintage, bottles, country)
- A system prompt instructing Claude to act as an expert sommelier, consider vintage and drinkability windows, return up to 10 ordered recommendations, and include one "Querschläger" (an unconventional but potentially great pick, placed mid-list). Each recommendation can also carry a "Drink now" label when the vintage is at or near its ideal drinking window.

Display the results as an ordered list in the overlay, with the Querschläger and "Drink now" labels visually distinguished. The user can dismiss when done.

### UI placement

```
┌──────────────────────────────────────┐
│ 42 bottles total         [Sommelier] │
└──────────────────────────────────────┘
```

The button is styled as a small pill, similar to the Sync button, on the right side of the KPI line. Only shown when a Claude API key is configured (same check as OCR).

### Implementation

1. Create `app/src/lib/sommelier/claude.ts`:
   - `runSommelierQuery(prompt: string, wines: Wine[], apiKey: string): Promise<SommelierResult>`
   - Uses same `API_URL` as OCR module; model: `claude-opus-4-6` (better nuance for pairing/drinkability)
   - Sends only **in-stock wines** (bottles > 0) as structured data: producer, name, type, vintage, bottles, country
   - System prompt instructs Claude to:
     - Act as an expert sommelier
     - Consider the full cellar inventory provided
     - Consider vintage and typical drinkability windows (is the wine ready, past peak, too young?)
     - Return up to 10 recommendations ordered by best pairing first
     - Include exactly one "Querschläger" pick: an unconventional/unexpected choice that could work surprisingly well — placed somewhere in the middle of the list (not #1 or #2), labeled as such
     - Mark recommendations "Drink now" when the vintage is at or near its ideal drinking window (based on type, region, and typical aging potential)
     - For each recommendation: wine name, producer, vintage, a one-line reason, querschläger flag, and drink-now flag
     - Reply with valid JSON: `{ "recommendations": [{ "producer": "...", "name": "...", "vintage": "...", "reason": "...", "querschlaeger": false, "drinkNow": false }, ...] }`
   - Parse the JSON response into a typed `SommelierResult`

2. Create `app/src/lib/components/SommelierModal.svelte`:
   - Text input + "Ask" button
   - Loading state while API call runs
   - Results rendered as an ordered list; the Querschläger entry gets a distinct label/badge; "Drink now" entries get a second label/badge
   - Each recommendation is tappable — links to the wine detail page (matched by producer + name + vintage)
   - Sommelier state (prompt + results) persisted in `sessionStorage` so navigating to a wine detail and back reopens the modal with results intact
   - Close/dismiss button (clears session state)
   - Error state (no API key, API error)

3. In `app/src/routes/+page.svelte`:
   - Add "Sommelier" button on the KPI line (right side)
   - Only visible when Claude API key is set
   - Opens `SommelierModal` with full `wines` array passed in
   - Load Claude API key on mount (reuse `loadClaudeApiKey` from settings)

### Acceptance criteria

- [ ] "Sommelier" button appears on KPI line when API key is configured
- [ ] Button hidden when no API key is set
- [ ] Tapping opens modal with text input
- [ ] Only in-stock wines (bottles > 0) sent to Claude API with the pairing query
- [ ] Claude considers vintage and drinkability window in its recommendations
- [ ] Up to 10 results displayed, ordered by best match
- [ ] Exactly one "Querschläger" entry in the middle of the list, visually labeled
- [ ] "Drink now" badge shown on wines at or near their ideal drinking window
- [ ] Loading spinner shown during API call
- [ ] API errors displayed gracefully
- [ ] Modal dismissible via close button
- [ ] Unit tests for `runSommelierQuery` (mock fetch, verify request shape and response parsing)
